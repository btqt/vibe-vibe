5.4.4 Tối ưu hóa code: Để AI giúp bạn cải thiện chất lượng code

Qua bài học này, bạn sẽ nắm được:

  • Khi nào thì nên tối ưu hóa code
  • 5 loại tối ưu hóa mà AI có thể giúp bạn
  • Ý thức cơ bản về rà soát code

Khi nào thì nên tối ưu hóa

Trước tiên nói về một nguyên tắc quan trọng:

Chức năng ổn định rồi hẵng tối ưu, đừng vừa viết tính năng vừa tối ưu.

Tại sao?

Bởi vì khi bạn đang thêm tính năng, code sẽ thay đổi liên tục. Lúc này mà tốn thời gian sắp xếp lại code, có thể tính năng tiếp theo lại phải sửa lại từ đầu.

Đợi đến khi các tính năng cốt lõi của danh sách việc cần làm đã hoàn thành, kiểm thử thông qua hết rồi, lúc đó hãy tính chuyện tối ưu.

Mục đích của tối ưu là:

  • Để code dễ đọc hơn (sau này chính bạn quay lại xem còn hiểu)
  • Giảm thiểu các vấn đề tiềm ẩn
  • Để code chạy nhanh hơn (đối với dự án nhỏ, cái này thường không phải trọng tâm)

5 loại tối ưu hóa mà AI có thể giúp bạn

Bạn không cần tự mình biết viết code xịn hơn, có thể để AI giúp bạn kiểm tra và cải thiện.

1. Kiểm tra lặp code

Hãy kiểm tra code của tôi, có logic nào bị lặp lại không?
Nếu có, hãy giúp tôi tách nó ra thành một hàm có thể tái sử dụng.

[Dán code của bạn]

Tại sao quan trọng: Code lặp lại nghĩa là nếu muốn sửa một logic nào đó, bạn phải sửa ở nhiều chỗ. Dễ bị sót, sinh ra bug.

2. Cải thiện đặt tên biến

Hãy kiểm tra cách đặt tên biến và hàm trong code của tôi, có chỗ nào chưa đủ rõ ràng không?
Hãy đưa ra gợi ý cải thiện, để code dễ đọc hơn.

[Dán code của bạn]

Ví dụ:

Trước cải thiện Sau cải thiện Lý do
a, b, x taskName, dueDate Tên có ý nghĩa, nhìn là biết cái gì
data todoList Cụ thể hơn
fn1 addNewTask Mô tả chức năng của hàm

3. Thêm chú thích tiếng Việt

Hãy thêm chú thích tiếng Việt cho đoạn code này, giải thích:
1. Mỗi hàm dùng để làm gì
2. Khối code then chốt đang xử lý logic gì
3. Những chỗ khó hiểu

[Dán code của bạn]

Tại sao quan trọng: Một tháng sau bạn xem lại code của chính mình, nếu không có chú thích có thể sẽ hoàn toàn không hiểu gì. Chú thích là viết cho chính mình trong tương lai.

4. Kiểm tra bug tiềm ẩn

Hãy kiểm tra code của tôi, có vấn đề hoặc bug tiềm ẩn nào không?
Đặc biệt chú ý:
1. Trường hợp biên có thể gây báo lỗi
2. Xử lý khi dữ liệu trống
3. Xác thực đầu vào của người dùng

[Dán code của bạn]

Ví dụ vấn đề thường gặp:

  • Người dùng chưa nhập gì đã bấm nút thêm
  • Hiển thị khi danh sách nhiệm vụ trống
  • Xử lý khi định dạng dữ liệu không đúng

5. Gợi ý tối ưu hiệu năng

Hãy kiểm tra code của tôi, có chỗ nào có thể tối ưu hiệu năng không?
Dự án của tôi là một danh sách việc cần làm, dữ liệu không lớn (nhiều nhất vài chục nhiệm vụ).
Chỉ đưa ra gợi ý tối ưu có ý nghĩa với quy mô này của tôi.

[Dán code của bạn]

Giải thích cho dự án nhỏ: Đối với dự án nhỏ như danh sách việc cần làm, hiệu năng thường không thành vấn đề. Đừng tối ưu quá mức (premature optimization). Nếu AI đưa ra gợi ý tối ưu quá phức tạp, có thể tạm thời chưa cần làm.

Một Prompt rà soát code hoàn chỉnh

Nếu bạn muốn AI thực hiện rà soát code toàn diện, có thể dùng mẫu này:

Hãy giúp tôi rà soát đoạn code này, đưa ra gợi ý cải tiến từ các góc độ sau:

1. Cấu trúc code: Có logic lặp lại không? Có thể đơn giản hóa thế nào?
2. Tính dễ đọc: Đặt tên biến đã rõ ràng chưa? Có cần thêm chú thích không?
3. Tính ổn định: Có trường hợp biên nào có thể gây lỗi không?
4. Tính an toàn: Có vấn đề an toàn nào rõ ràng không?

Hãy sắp xếp gợi ý theo mức độ ưu tiên (nói cái quan trọng nhất trước), và đưa ra phương án sửa đổi cụ thể.

Ý thức cơ bản về rà soát code

Ở bài 5.3 chúng ta đã nhắc đến, code do AI tạo ra không phải lúc nào cũng an toàn.

Theo báo cáo năm 2025 của Veracode, 45% code do AI tạo ra không vượt qua được kiểm tra an toàn.

Điều này không có nghĩa là AI không tốt, mà là: Bạn cần có ý thức rà soát cơ bản đối với code AI đưa ra.

Khi rà soát hãy chú ý những điểm này

Điểm chú ý Nội dung kiểm tra
An toàn dữ liệu Trong code có lộ lọt thông tin nhạy cảm (mật khẩu, khóa API) không
Xác thực đầu vào Dữ liệu người dùng nhập vào có được kiểm tra không
Xử lý lỗi Khi xảy ra lỗi có bị crash không, có đưa ra lời nhắc cho người dùng không
Chức năng đúng đắn Logic AI thực hiện có nhất quán với mong đợi của bạn không

Nếu đọc code không hiểu lắm cũng không sao. Ít nhất hãy làm được:

  • Chạy thử nghiệm, xem chức năng có bình thường không
  • Xem các phần then chốt, ví dụ chỗ xử lý đầu vào của người dùng
  • Có thắc mắc thì hỏi AI: "Đoạn code này đang làm gì? Có vấn đề gì không?"

Tối ưu không bằng viết lại

Lời nhắc quan trọng cuối cùng:

Tối ưu là cải tiến từng bước nhỏ trên nền code hiện có, không phải đập đi xây lại.

Nếu AI gợi ý bạn "dùng khung X viết lại toàn bộ dự án", thì đó thường không phải là điều bạn cần làm lúc này.

Đối với dự án giai đoạn học tập, giữ cho code hiện tại chạy được là tốt rồi. Đợi bạn làm nhiều dự án hơn, tích lũy nhiều kinh nghiệm hơn, tự nhiên sẽ biết khi nào nên dùng giải pháp tốt hơn.
