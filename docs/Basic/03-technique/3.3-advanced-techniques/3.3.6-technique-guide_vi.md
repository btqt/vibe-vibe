# 3.3.6 Hướng dẫn lựa chọn kỹ thuật: Trường hợp nào dùng kỹ thuật gì

## Sau bài học này, bạn sẽ nắm được

- Phán đoán nhanh xem nhiệm vụ nên dùng loại kỹ thuật nào
- Chiến lược sử dụng kết hợp các kỹ thuật
- Một bảng tra cứu nhanh có thể xem bất cứ lúc nào

## Quy trình ra quyết định cốt lõi

Gặp một nhiệm vụ, hãy chọn kỹ thuật theo quy trình sau:

```
Bắt đầu
  ↓
Nhiệm vụ này AI có nên "biết làm ngay" không?
  ├─ Có → Zero-shot (Hỏi trực tiếp)
  └─ Không ↓

Có cần định dạng đầu ra cụ thể không?
  ├─ Có → Few-shot (Cho ví dụ)
  └─ Không ↓

Có cần suy luận/phân tích nhiều bước không?
  ├─ Có → Chain of Thought (Suy nghĩ phân bước)
  └─ Không ↓

Có cần so sánh nhiều phương án không?
  ├─ Có → Tree of Thoughts (So sánh đa đường dẫn)
  └─ Không ↓

Chất lượng đầu ra có quan trọng không?
  ├─ Có → Thêm Self-Critique (Tự kiểm tra sửa lỗi)
  └─ Không → Zero-shot là đủ rồi
```

## Bảng quyết định hoàn chỉnh

| Loại nhiệm vụ             | Kỹ thuật đề xuất | Lý do                                  | Nhiệm vụ ví dụ                         |
| :------------------------ | :--------------- | :------------------------------------- | :------------------------------------- |
| Chuyển đổi định dạng code | Zero-shot        | AI đã nắm quy tắc chuyển đổi           | Python 2 → Python 3                    |
| Tạo nội dung              | Zero-shot        | Nhiệm vụ sáng tạo không cần ví dụ      | Viết mô tả sản phẩm                    |
| Giải thích code           | Zero-shot        | AI có khả năng hiểu code tốt           | Giải thích đoạn code này làm gì        |
| Đầu ra định dạng hóa      | Few-shot         | Dùng ví dụ đảm bảo định dạng nhất quán | Tạo tài liệu API                       |
| Nhiệm vụ phân loại        | Few-shot         | Dùng ví dụ làm rõ tiêu chuẩn phân loại | Phân loại phản hồi người dùng          |
| Mô phỏng phong cách       | Few-shot         | Dùng ví dụ học phong cách              | Viết văn án theo phong cách cụ thể     |
| Phân tích độ phức tạp     | Chain of Thought | Cần suy luận phân bước                 | Phân tích độ phức tạp thời gian        |
| Rà soát Bug               | Chain of Thought | Cần suy luận từng bước                 | Phân tích tại sao trang không cập nhật |
| Thẩm định logic           | Chain of Thought | Cần xem xét nhiều nhánh                | Thẩm định logic xác thực form          |
| Lựa chọn kỹ thuật         | Tree of Thoughts | Cần so sánh nhiều phương án            | Chọn cơ sở dữ liệu                     |
| Quyết định kiến trúc      | Tree of Thoughts | Cần cân nhắc nhiều phương án           | Thiết kế phong cách API                |
| Code quan trọng           | + Self-Critique  | Cần đảm bảo chất lượng                 | Logic nghiệp vụ cốt lõi                |
| Đầu ra công khai          | + Self-Critique  | Cần giảm thiểu sai sót                 | Tài liệu API đối ngoại                 |

## Chiến lược sử dụng kết hợp

Kỹ thuật đơn lẻ đôi khi chưa đủ, có thể sử dụng kết hợp:

### Kết hợp 1: Few-shot + Self-Critique

**Bối cảnh**: Cần đầu ra định dạng hóa, và yêu cầu chất lượng cao

```
Nhiệm vụ: Tạo tài liệu giao diện API

Ví dụ 1:
Giao diện: GET /users
Tài liệu:
- Mô tả: Lấy danh sách người dùng
- Tham số: page (phân trang), limit (số lượng mỗi trang)
- Trả về: Mảng người dùng

Ví dụ 2:
Giao diện: POST /users
Tài liệu:
- Mô tả: Tạo người dùng mới
- Tham số: name (bắt buộc), email (bắt buộc)
- Trả về: Đối tượng người dùng mới tạo

Vui lòng tạo tài liệu cho giao diện sau:
Giao diện: DELETE /users/:id

Sau khi tạo xong, vui lòng tự kiểm tra:
- Có tuân thủ định dạng ví dụ không?
- Có bỏ sót thông tin cần thiết không?
Nếu có vấn đề hãy sửa lại.
```

### Kết hợp 2: Chain of Thought + Self-Critique

**Bối cảnh**: Nhiệm vụ phân tích phức tạp, cần đảm bảo suy luận đúng đắn

```
Vui lòng phân tích độ phức tạp thời gian của thuật toán sắp xếp này.

Các bước phân tích:
1. Nhận diện cấu trúc vòng lặp
2. Phân tích số lần thực thi của mỗi tầng vòng lặp
3. Tính toán tổng độ phức tạp thời gian

Sau khi phân tích xong, vui lòng tự kiểm tra:
- Suy luận của mỗi bước có đúng không?
- Kết luận cuối cùng có hợp lý không?
Nếu có lỗi hãy sửa lại.
```

### Kết hợp 3: Tree of Thoughts + Chain of Thought

**Bối cảnh**: Quyết định phức tạp, mỗi phương án đều cần phân tích sâu

```
Vui lòng chọn phương án quản lý trạng thái cho ứng dụng của tôi.

Phân tích 3 phương án:
1. React Context
2. Redux Toolkit
3. Zustand

Đối với mỗi phương án, hãy phân tích phân bước:
- Bước 1: Hiểu nguyên lý cốt lõi của nó
- Bước 2: Phân tích ưu thế trong bối cảnh của tôi
- Bước 3: Phân tích vấn đề tiềm ẩn
- Bước 4: Đưa ra bối cảnh áp dụng

Cuối cùng so sánh và đề xuất.
```

## Tham khảo tần suất sử dụng hàng ngày

Dựa trên kinh nghiệm phát triển thực tế, tần suất sử dụng các kỹ thuật đại khái như sau:

| Kỹ thuật         | Tần suất dùng hàng ngày | Bối cảnh điển hình                   |
| :--------------- | :---------------------- | :----------------------------------- |
| Zero-shot        | 50%                     | Đa số nhiệm vụ đơn giản              |
| Few-shot         | 25%                     | Đầu ra định dạng hóa                 |
| Chain of Thought | 15%                     | Phân tích logic phức tạp             |
| Self-Critique    | 8%                      | Đảm bảo chất lượng đầu ra quan trọng |
| Tree of Thoughts | 2%                      | Quyết định lựa chọn kỹ thuật         |

**Đề xuất thực tiễn**:

1.  Bắt đầu thử từ Zero-shot
2.  Nếu đầu ra không ưng ý, nâng cấp lên kỹ thuật khác
3.  Đầu ra quan trọng nhớ thêm Self-Critique

## Danh sách lựa chọn nhanh

### Tôi muốn chuyển đổi code/chuyển đổi định dạng

→ **Zero-shot**

### Tôi cần định dạng đầu ra thống nhất

→ **Few-shot** (Cho 2-3 ví dụ)

### Tôi cần phân tích logic phức tạp/rà soát Bug

→ **Chain of Thought** (Suy nghĩ phân bước)

### Tôi cần chọn kỹ thuật/so sánh phương án

→ **Tree of Thoughts** (So sánh đa phương án)

### Đầu ra này rất quan trọng, không được sai

→ Thêm **Self-Critique** (Tự kiểm tra sửa lỗi)

## Thẻ tra cứu nhanh kỹ thuật bài này

Có thể lưu lại hoặc in ra, tra cứu bất cứ lúc nào:

```
┌─────────────────────────────────────────────────────┐
│              Thẻ tra cứu kỹ thuật Prompt v1.0         │
│                                                     │
├─────────────────────────────────────────────────────┤
│                                                     │
│  Zero-shot    │ Nhiệm vụ AI biết làm, hỏi ngay      │
│               │ VD: Chuyển đổi code, tạo nội dung   │
│                                                     │
│  Few-shot     │ Cần định dạng cụ thể, cho ví dụ     │
│               │ VD: Tài liệu API, nhiệm vụ phân loại│
│                                                     │
│  CoT          │ Suy luận phức tạp, suy nghĩ phân bước│

│  ToT          │ So sánh phương án, khám phá đa đường │

└─────────────────────────────────────────────────────┘
```

## Kết nối với bài tiếp theo

Chúc mừng bạn, đã hoàn thành việc học phần 3.3 Kỹ thuật Prompt nâng cao!

Bây giờ bạn đã nắm vững:

- **3.1**: Nên nói cho AI thông tin gì (Ngữ cảnh)
- **3.2**: Tổ chức thông tin thế nào (Khung cấu trúc hóa)
- **3.3**: Dùng cách thức nào để AI xử lý (Kỹ thuật Prompt)

Bài tiếp theo **3.4 Viết bản PRD đầu tiên của bạn**, chúng ta sẽ **tích hợp** tư duy sản phẩm của Chương 2 và kỹ thuật Prompt của Chương 3, hình thành một quy trình làm việc hoàn chỉnh.

Bạn sẽ học được:

- Cách chuyển hóa kết quả của 3 câu hỏi linh hồn, chân dung người dùng, tư duy phép trừ thành PRD
- Một mẫu PRD tinh gọn phù hợp cho người mới bắt đầu
- Cách dùng độ ưu tiên dẫn dắt AI phát triển phân giai đoạn

## Trọng điểm bài học

✓ **Quy trình chọn kỹ thuật cốt lõi**: Biết làm ngay → Cần định dạng → Cần suy luận → Cần so sánh → Cần chất lượng

✓ **Tổ hợp thường dùng nhất**: Few-shot + Self-Critique, CoT + Self-Critique

✓ **Tần suất dùng hàng ngày**: Zero-shot 50%, Few-shot 25%, CoT 15%, Self-Critique 8%, ToT 2%

✓ **Đề xuất thực tiễn**: Bắt đầu từ Zero-shot, không ưng thì nâng cấp; đầu ra quan trọng thêm Self-Critique

## Danh sách kiểm tra hoàn thành chương này

Trước khi bước vào bài tiếp theo, hãy xác nhận bạn đã:

- [ ] Hiểu Zero-shot: Hỏi trực tiếp nhiệm vụ AI biết làm
- [ ] Hiểu Few-shot: Dùng ví dụ dạy AI định dạng
- [ ] Hiểu Chain of Thought: Suy luận phức tạp phân bước suy nghĩ
- [ ] Biết sự tồn tại của Tree of Thoughts: Dùng khi so sánh phương án
- [ ] Hiểu Self-Critique: Đầu ra quan trọng để AI tự kiểm tra
- [ ] Có thể chọn kỹ thuật phù hợp dựa trên loại nhiệm vụ

Nếu đã tích hết các mục trên, chúc mừng bạn, sẵn sàng bước vào bài 3.4 rồi!
